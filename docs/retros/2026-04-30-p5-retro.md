## Lessons learned

### 1. drizzle-orm mutates postgres-js serializers (the headline finding)

The single biggest cost-of-the-phase. `drizzle-orm/postgres-js/driver.js:16-22` â€” when you call `drizzle(sql)`, drizzle MUTATES `client.options.serializers["114"]` and `["3802"]` (json + jsonb OIDs) to identity passthroughs so its own type system can handle JSON encoding. After this mutation, raw postgres-js queries against the same client receive raw `Object`/`Array` at the wire-bytes layer (`b.str(x)` â†’ `Buffer.byteLength(x)`), which Node 22 rejects with `TypeError: Received an instance of Array/Object`.

In our codebase: `packages/db/src/client.ts:13` exports both `sql = postgres(...)` AND `db = drizzle(sql)` â€” same instance, mutated. `privilegedSql` is NOT fed to drizzle â€” its default `JSON.stringify` jsonb serializer survives.

**Symptoms that wasted ~6 fix iterations on PR #9:**
- Test seeds via `privilegedSql\`${arr}\`` worked (no drizzle on client) â†’ false signal that bare-value form works
- `audit_log_payload_object` CHECK violated by `${JSON.stringify(payload)}::jsonb` (single-cast double-encodes under privilegedSql) â†’ fixed by switching writer to bare `${object}`
- Bare `${object}` then failed under `sql.begin â†’ tx` route handlers (drizzle-mutated client) â†’ TypeError
- Single-cast then failed in writer (privilegedSql double-encodes again)

**Fix that survives both clients:** `${JSON.stringify(value)}::text::jsonb`. The double cast pins the wire type to TEXT (postgres OID 25), whose serializer survives drizzle's mutation as a no-op on strings. Postgres casts text â†’ jsonb server-side. Verified empirically against both `sql.begin` and `privilegedSql.begin` contexts.

**Files using the double-cast pattern (correct):** `apps/api/src/routes/mapping-rules.ts`, `packages/db/src/audit-log.ts`.

**Files still using single-cast (silent latent bug, P6 cleanup):** `packages/db/src/chain.ts:133`. It works today because the chain is only written from `sql.begin` (drizzle-mutated, identity is a no-op on the JSON string). If anything ever writes the chain from `privilegedSql`, events will store as jsonb scalar strings.

**Mitigation going forward:** the JSDoc on `insertAuditLog` is the canonical documentation. New jsonb writers should reference it.

### 2. The `audit_log` RLS positive-control test pattern works

Task 2.1's mandate: "FIRM_A session cannot read FIRM_B rows" + "GUC unset â†’ query returns no rows" must both pass before Task 2.4's writer goes live. They did. The test pattern (in `apps/api/src/routes/audit-log.test.ts`) is now the template for any future firm-scoped table â€” paste it, swap the table name and FK, and you've inherited the keystone.

The "GUC unset â†’ fail-safe deny" assertion specifically caught a thinko: NULLIF wrapping in the policy is essential because `current_setting('app.current_firm_id', true)::uuid` on an unset GUC returns `''` (empty string), and `'' = firm_id` is FALSE for every uuid â€” but only because of luck. If postgres ever changes that coercion, every firm leaks. NULLIF makes it explicit: `NULLIF(current_setting(...), '')::uuid` returns NULL, and `NULL = firm_id` is NULL (which evaluates as FALSE in WHERE/USING/WITH CHECK). The migration mirrors `0003`'s pattern.

### 3. Cross-swimlane drift catch â€” `claim-pdf.ts` `manual â†’ INVOICE` should have been `RECEIPT`

Task 6.2's job was to add inline comments documenting the convention. While writing those comments, the agent noticed `claim-pdf.ts` (shipped from PR #6 / swimlane C) had `manual â†’ INVOICE` whereas `docs/decisions/0006-p4-merge-plan.md Â§4.1` says `manual â†’ RECEIPT`. ### 4. CI flake economics: admin-merge with documentation

Three of the four P5 PRs hit pre-existing flakes in main-side code (subject-tenants ACL test 445, `findOrCreateUser` race-window test 23, `chain.test.ts` `audit_score_snapshot` FK cleanup error). The flakes are in non-P5 code, hit main itself, and surface differently each rerun. Admin-merge with a comment explaining the diff (this PR's changes vs the failing tests) was the right call â€” same precedent as PR #6.

The flakes are themselves a P6 followup item: hardening the cleanup hooks to handle interlocked FKs, and replacing the race-window test with a deterministic version.

### 5. Agent prompts should encode hard-won knowledge as constraints

P5c was dispatched AFTER P5b's debugging journey concluded. The dispatch prompt explicitly stated:
- "Use `${JSON.stringify(value)}::text::jsonb` double-cast for ANY jsonb binding under `sql.begin â†’ tx` (drizzle mutates serializers â€” see audit-log.ts JSDoc)"
- "DO NOT modify `chain.ts` (out of scope; P6 cleanup item)"

P5c shipped a clean PR with zero CI failures on the first run. Compare with P5b's 9-iteration cycle without that knowledge. The cost difference is real â€” encoding lessons-learned as agent constraints upfront saves a debugging cycle per agent.

### 6. Test seeds and route handlers behave differently on the same SQL

Test seeds use `privilegedSql\`...\`` (cpa role, RLS bypass, default postgres-js serializers). Route handlers use `sql.begin â†’ tx` (cpa_app role, RLS-aware, drizzle-mutated serializers). Same SQL string can succeed in one and fail in the other. **Future bug-hunting heuristic:** when "the route fails but the test seed works," check the client. Don't assume identical behavior.
