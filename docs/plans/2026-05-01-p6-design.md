## Section 1 — Architecture overview

P6 layers three velocity-tiered AI agents on top of the event chain. Each agent has its own wiring pattern that matches its natural cadence.

```
                    ┌────────────────── EVENT CHAIN ──────────────────┐
                    │                                                  │
                    │  EXPENDITURE_INGESTED ──► [A] ──► EXPENDITURE_   │
                    │                            (pg-boss,  CLASSIFIED │
                    │                             auto-batch)          │
                    │                                                  │
                    │  HYPOTHESIS / DESIGN /                           │
                    │  EXPERIMENT / OBSERVATION ─► [B] ─► ACTIVITY_    │
                    │  / NEW_KNOWLEDGE / etc.    (pg-boss, REGISTER_   │
                    │                             manual)  DRAFTED     │
                    │                                                  │
                    │  + activity finalised ─►   [C] ──► NARRATIVE_    │
                    │                            (HTTP +   DRAFTED     │
                    │                             SSE)    (metadata    │
                    │                                      only)       │
                    └──────────────────────────────────────────────────┘

   Storage:  narrative_draft + narrative_draft_version (Agent C only —
             A and B encode their full output in event payloads)
```

Three latency tiers, three wiring patterns:

| Agent | Trigger | Backend | Latency | UX surface |
| --- | --- | --- | --- | --- |
| A — Expenditure classifier | New `EXPENDITURE_INGESTED` arrives | `pg-boss` job, batched per minute | Background (minutes) | Confidence badges + suggested activity in existing expenditure UI |
| B — Activity register synthesizer | Consultant clicks "Synthesize register" on a project | `pg-boss` job, single-run | 30–90s async | Loading state → "Review draft register" page |
| C — Narrative drafter | Consultant clicks "Generate narrative" or "Regenerate this section" | HTTP route returning SSE stream | Streaming (5–30s per section) | Real-time text appearing in editor; per-section regen buttons |

**Why three patterns instead of one:** Agent A wants to run continuously as expenditures land — making it interactive would waste consultant attention on something that should just happen. Agent B is consequential (it shapes the whole submission structure) so consultants want explicit control over when it runs. Agent C is interactive prose — the consultant is editing live, so streaming + regen is the right UX.

**Events vs tables split:** A and B emit events (`EXPENDITURE_CLASSIFIED`, `ACTIVITY_REGISTER_DRAFTED`); the chain stays the source of truth and the UI reads the latest event of each kind. C uses tables because narratives are big, segmented, versioned, and edited; events would bloat the chain. C still emits a `NARRATIVE_DRAFTED` marker event with `{ draft_id, content_hash, segment_count }` so auditors can verify persistence integrity from the chain.

### Two new tables (Agent C storage)

```sql
-- 0029_narrative_draft.sql
CREATE TABLE narrative_draft (
  tenant_id        uuid NOT NULL,
  id               uuid NOT NULL,
  activity_id      uuid NOT NULL,
  section_kind     text NOT NULL,
    -- 'new_knowledge' | 'hypothesis' | 'uncertainty' | 'experiments_and_results'
  current_version  integer NOT NULL,
  status           text NOT NULL,
    -- 'streaming' | 'complete' | 'accepted' | 'archived'
  segments         jsonb NOT NULL,  -- list of NarrativeSegment for THIS section
  content_hash     text NOT NULL,
  model            text NOT NULL,
  prompt_version   text NOT NULL,
  idempotency_key  text,
  created_at       timestamptz NOT NULL DEFAULT now(),
  updated_at       timestamptz NOT NULL DEFAULT now(),
  created_by_user_id uuid NOT NULL,
  PRIMARY KEY (tenant_id, id),
  UNIQUE (tenant_id, activity_id, section_kind),
  CONSTRAINT narrative_draft_section_kind_valid CHECK (
    section_kind IN ('new_knowledge','hypothesis','uncertainty','experiments_and_results')
  ),
  CONSTRAINT narrative_draft_status_valid CHECK (
    status IN ('streaming','complete','accepted','archived')
  )
  -- FK to activity, RLS by tenant_id
);

-- 0030_narrative_draft_version.sql
-- Append-only history. Section regen creates a new row, then bumps narrative_draft.current_version.
CREATE TABLE narrative_draft_version (
  tenant_id        uuid NOT NULL,
  id               uuid NOT NULL,
  draft_id         uuid NOT NULL,         -- FK to narrative_draft.id
  version          integer NOT NULL,
  segments         jsonb NOT NULL,
  content_hash     text NOT NULL,
  model            text NOT NULL,
  prompt_version   text NOT NULL,
  parent_version   integer,                -- regen lineage; null on initial
  generation_kind  text NOT NULL,          -- 'initial' | 'section_regen' | 'edit'
  created_at       timestamptz NOT NULL DEFAULT now(),
  created_by_user_id uuid NOT NULL,
  PRIMARY KEY (tenant_id, id),
  UNIQUE (tenant_id, draft_id, version),
  CONSTRAINT narrative_draft_version_generation_kind_valid CHECK (
    generation_kind IN ('initial','section_regen','edit')
  )
);
```

Both tables use the standard tenant-isolated RLS pattern from P5 (`FORCE ROW LEVEL SECURITY`, `NULLIF`-wrapped USING/WITH CHECK clauses, GRANT to `cpa_app`). `narrative_draft` gets `SELECT, INSERT, UPDATE, DELETE` (consultants edit drafts); `narrative_draft_version` gets `SELECT, INSERT` only (append-only — same pattern as `audit_log`).

## Section 3 — Agent A: expenditure classifier

**Job:** read each `EXPENDITURE_INGESTED` event, classify the expenditure as R&D-eligible / ineligible / needs-review, and (if eligible) suggest which `activity_id` to map it to. Emit `EXPENDITURE_CLASSIFIED`.

### Inputs (per call)

- The expenditure row: `vendor_name`, `description`, `total_amount`, `currency`, `expenditure_date`, `source` (xero_invoice / xero_bank_tx / xero_receipt / manual), `kind` (INVOICE / BANK_TX / RECEIPT — manual maps to RECEIPT per `claim-pdf.ts` convention from PR #6 / P5 retro)
- Project context: `project_name`, `industry_sector`, `claim.fiscal_year`
- Current activity register (if drafted): each activity's `name`, `kind`, `statutory_anchor`, brief description
- Up to 10 most-recent R&D evidence events on the same project — gives semantic context

### Output (tool-use schema)

```ts
{
  expenditure_id: Uuid,
  decision: 'eligible' | 'ineligible' | 'needs_review',
  eligibility_probability: number,             // 0..1
  statutory_anchor: 's.355-25' | 's.355-30' | 'ineligible',
  suggested_activity_id: Uuid | null,
  rationale: string,                           // 1-3 sentences for hover-card
  uncertainty_reason: string | null            // populated when decision='needs_review'
}
```

Emitted via Anthropic tool-use mode (no free-text JSON parsing; reuses `@cpa/agents/runtime/anthropic-client.ts`).

### Confidence handling

| `eligibility_probability` | Behavior |
| --- | --- |
| `≥ 0.85` and `decision != 'needs_review'` | Auto-applied; expenditure UI shows green badge "Auto-classified — review optional" |
| `0.70 - 0.85` | Auto-applied with amber badge "Review recommended" |
| `< 0.70` OR `decision='needs_review'` | Force-set to `needs_review`; red badge + `uncertainty_reason`; consultant must classify manually before claim inclusion |

Thresholds live in `packages/agents/src/classifier-expenditure/thresholds.ts`. Calibrated after Phase 1 dogfood telemetry.

### Model + cost

**Haiku 4.5.** ~$0.10 per 1000-expenditure claim (200 input + 100 output tokens × $0.25/$1.25 per M). Sonnet would 4-5× the cost without measurably better classification on this task. Override via `EXPENDITURE_CLASSIFIER_MODEL` env var.

### Idempotency + job wiring

`idempotency_key = sha256(expenditure_id + prompt_version + model + input_bundle_hash)`.

`pg-boss` queue `expenditure-classify`. Two trigger paths:

1. **Auto on ingest:** event-insertion hook (application layer in `xero-accounting-sync.ts` and the manual-expenditure POST handler) enqueues a job. Not a DB trigger — keeps chain insertion pure.
2. **Manual re-run:** `POST /v1/expenditures/:id/reclassify` (admin/consultant) for cases where the consultant edits the expenditure or the prompt rolls out.

Batch size: 25 expenditures per job invocation. Smaller batches give better progress visibility and lower variance under model errors. Tunable via `P6_AGENT_A_BATCH_SIZE`.

### Consultant override path

Existing `OVERRIDE` event mechanism from P2 already handles re-classification:

- Consultant sets decision manually in UI
- UI POSTs `kind: OVERRIDE`, `override_of_event_id` → original `EXPENDITURE_CLASSIFIED`, `override_new_kind: EXPENDITURE_CLASSIFIED` with corrected payload
- Existing chain mechanics handle supersedence

No new override surface. **One of the elegant payoffs of the P2 chain design — agents and humans use the same primitive to disagree.**

### Failure modes

- **Anthropic 5xx / timeout:** `pg-boss` retries (1min, 5min, 30min). After exhaustion, emit `EXPENDITURE_CLASSIFIED` with `decision='needs_review'`, `uncertainty_reason='classifier unavailable'`. Consultant sees red badge.
- **Stub fallback (`CLASSIFIER_IMPL=stub`):** deterministic regex (vendor_name "Atlassian" → ineligible; "AWS" → eligible; etc.). CI runs without API keys.
- **Empty register (B hasn't run yet):** `suggested_activity_id: null`. Consultant assigns manually post-B.
- **No project context:** `decision='needs_review'`, `uncertainty_reason='not yet attached to a project'`.

## Section 4 — Agent B: activity register synthesizer

**Job:** read R&D evidence events for a project, propose how to GROUP them into named activities, classify each as core (s.355-25) or supporting (s.355-30), emit `ACTIVITY_REGISTER_DRAFTED`.

### Inputs

```ts
{
  project: { id, name, industry_sector, started_at, fiscal_year },
  events: CompressedEvent[],                  // capped at 200 most-recent R&D evidence kinds
  existing_activities: AcceptedActivity[]      // empty on first run; populated on re-run
}
```

`CompressedEvent = { id, kind, captured_at, summary }` where `summary` is a 50-word extract from `payload.text`. Keeps input bundle under ~80K tokens for typical projects.

`AcceptedActivity` shows existing `activity` rows the consultant has already created. The agent is told **"do not propose to recluster events that are already in an accepted activity — only propose new activities for unclustered events."** This makes re-runs additive and non-destructive.

### Output (tool-use schema)

```ts
{
  proposed_activities: ProposedActivity[],
  unclustered_event_ids: Uuid[],              // events the model couldn't confidently place
  total_input_events: number,
  events_truncated: boolean,
  synthesizer_notes: string                   // 1-2 paragraph rationale for consultant
}

ProposedActivity = {
  proposed_id: Uuid,                           // ephemeral — replaced when consultant accepts
  name: string,                                // 5-12 words
  kind: 'core' | 'supporting',
  statutory_anchor: 's.355-25' | 's.355-30',
  rationale: string,                           // 1 paragraph
  clustered_event_ids: Uuid[],
  confidence: number,                          // 0..1
  proposed_hypothesis: string | null,          // pre-fill for Agent C's hypothesis section
  proposed_uncertainty: string | null
}
```

The `proposed_hypothesis` + `proposed_uncertainty` fields are bonus output — Agent C inherits a head-start when present. If null, Agent C derives from events directly.

### Clustering approach (pure LLM for MVP)

Single Sonnet call with the full input bundle. No algorithmic pre-clustering (embeddings, k-means, time-windows) in MVP. Reasons:

- Small-to-medium projects (≤200 events) fit in one Sonnet context window
- Sonnet's semantic clustering on R&D narrative is strong out of the box; embedding clusters tend to over-split or merge across topic boundaries
- Pre-clustering is a P7 task, purely additive (slots in front of the LLM call)

Hard cap: 200 input events. If exceeded, agent runs on the most-recent 200; response sets `events_truncated: true`; UI surfaces a "consider running on subsets" warning.

### Model + cost

**Sonnet 4.5.** ~80K input + ~2-4K output tokens × $3/$15 per M → ~$0.30 per synthesis. Volume is low (1-10 synthesis calls per project per fiscal year). Override via `ACTIVITY_REGISTER_SYNTHESIZER_MODEL`.

### Idempotency + job wiring

`idempotency_key = sha256(project_id + sorted_event_ids + prompt_version + model)`. Re-runs with identical input dedupe.

`pg-boss` queue `activity-register-synthesize`. Single-shot job triggered by `POST /v1/projects/:id/activity-register/synthesize` (admin/consultant role). UI polls `/v1/projects/:id/activity-register/latest` for completion (~30-90s).

### Acceptance flow (proposed → real activities)

Proposed activities are not real `activity` rows yet. Consultant reviews:

```
POST /v1/projects/:id/activity-register/accept
body: {
  proposed_ids: Uuid[],
  edits: Record<proposed_id, {
    name?, kind?, statutory_anchor?, rationale?
  }>
}
```

For each accepted `proposed_id`:

1. Create real `activity` row with (possibly edited) fields
2. Emit `ACTIVITY_CREATED` event (existing kind from P4)
3. The original `clustered_event_ids` are NOT moved or relinked — events stay where they are; the activity is just created. Linkage is by claim/project.

Rejected proposals are dropped silently (no `ACTIVITY_REJECTED` event). Next B run, those events are unclustered again — they may cluster differently as more events arrive.

### Re-runnability + supersedence

Each `ACTIVITY_REGISTER_DRAFTED` event is independent. UI shows the latest one as canonical. **No `OVERRIDE` mechanism here** — re-runs simply emit a new event; consultants can compare versions. This is intentionally lighter than per-expenditure override because synthesis is consequential and consultants will want to consciously re-trigger it.

### Failure modes

- **<5 events on project:** emit `ACTIVITY_REGISTER_DRAFTED` with `proposed_activities: []`, `synthesizer_notes: "Insufficient evidence (5 minimum)."`
- **>200 events:** process 200 most-recent, set `events_truncated: true`
- **Anthropic 5xx:** `pg-boss` retries (1min, 5min, 30min). On final failure, emit no event; endpoint returns 503 with `requestId`.
- **Stub fallback:** group by `subject_tenant_id` and ISO week; one proposed activity per (subject_tenant, week); name `"Activity for week of <date>"`; kind=`core`, anchor=`s.355-25`. CI + dev machines run end-to-end without Anthropic.

## Section 5 — Agent C: narrative drafter (streaming)

**Job:** generate AusIndustry narrative for an activity, structured as δ-hybrid segments, streamed to the consultant in real-time, persisted with per-section version history. Supports re-rolling individual sections.

### The 4 sections per activity

| `section_kind` | What goes there |
| --- | --- |
| `new_knowledge` | New knowledge being sought; why not deducible from existing knowledge |
| `hypothesis` | Hypothesis tested; predicted outcome |
| `uncertainty` | Sources of uncertainty at start (could not be known/deduced in advance) |
| `experiments_and_results` | Experimental activities undertaken; observed results; how they refined/refuted hypothesis |

One activity → 4 rows in `narrative_draft` (one per section_kind), plus `narrative_draft_version` rows.

### Inputs (per generation request)

```ts
{
  activity: { id, name, kind, statutory_anchor, project_id },
  project: { id, name, industry_sector, fiscal_year },
  clustered_events: CompressedEvent[],         // events the consultant accepted as clustering this activity
  prefill: {                                    // from Agent B if available
    proposed_hypothesis?: string,
    proposed_uncertainty?: string
  } | null,
  existing_sections: Record<section_kind, NarrativeSegment[]> | null,
    // populated only on section regen — gives model context for consistency
  target_section_kinds: section_kind[]
    // ['new_knowledge','hypothesis','uncertainty','experiments_and_results'] on initial gen
    // single-element on section regen
}
```

### Output via streaming tool-use

Single tool exposed to the model:

```ts
emit_segment({
  section_kind: 'new_knowledge' | 'hypothesis' | 'uncertainty' | 'experiments_and_results',
  segment_index: number,                        // 0-based ordinal within the section
  type: 'prose' | 'claim',
  text: string,                                  // 1-3 sentences typically
  citing_events?: Uuid[]                         // REQUIRED if type='claim', forbidden if type='prose'
})
```

Model calls `emit_segment` repeatedly. Anthropic's tool-use streaming protocol gives clean boundaries: each `tool_use` content block is one full call. As each call's JSON completes, server validates → flushes to SSE → accumulates into draft buffer.

### Server-side validation (per-segment, on each tool call)

```ts
function validateSegment(seg: Segment, clusteredEventIds: Set<Uuid>): SegmentValidation {
  if (seg.type === 'claim' && (!seg.citing_events || seg.citing_events.length === 0)) {
    return { ok: false, reason: 'claim segment missing citing_events' };
  }
  if (seg.type === 'prose' && seg.citing_events && seg.citing_events.length > 0) {
    return { ok: false, reason: 'prose segment has unexpected citing_events' };
  }
  if (seg.type === 'claim') {
    const valid = seg.citing_events.every(id => clusteredEventIds.has(id));
    if (!valid) return { ok: false, reason: 'cites event outside this activity' };
  }
  return { ok: true };
}
```

On validation failure:

1. Server logs rejected segment + reason (telemetry — track how often the model gets this wrong)
2. Server does NOT flush bad segment to client
3. Server sends correction prompt back to model: *"Your segment_index N for section X failed validation: <reason>. Re-emit it correctly."*
4. Up to 2 correction retries before giving up. After: emit segment as `prose` (downgraded), tag draft with `validation_downgraded_count: N`; UI shows yellow warning.

This validate-and-correct loop is what makes the δ hybrid actually robust.

### SSE protocol

```
event: start
data: {"draft_id":"<uuid>","version":3,"target_sections":[...]}

event: segment
data: {"section_kind":"new_knowledge","segment_index":0,"type":"prose","text":"Under s.355-25..."}

event: segment
data: {"section_kind":"new_knowledge","segment_index":1,"type":"claim","text":"In sprint 3...","citing_events":["<uuid>","<uuid>"]}

event: section_complete
data: {"section_kind":"new_knowledge","segment_count":7,"claim_count":4}

event: done
data: {"draft_id":"<uuid>","version":3,"content_hash":"<sha256>","total_segments":28,"total_claims":17,"narrative_drafted_event_id":"<uuid>"}

event: error
data: {"reason":"anthropic_5xx","retryable":true,"partial_segments_persisted":true}
```

Client appends segments to per-section builder; on `section_complete` the section is renderable; on `done` the full draft is persistable. On `error` with `partial_segments_persisted: true`, the next request can resume.

### Section regen mechanics

```
POST /v1/activities/:id/narrative/sections/:section_kind/regenerate
```

Server flow:

1. Load current `narrative_draft` for the activity
2. Build prompt input including `existing_sections` for the OTHER 3 sections (rendered as plain text — "these have been written and accepted; do not contradict them")
3. Set `target_section_kinds = [requested_section]`
4. Stream as above; model only emits segments for the requested section
5. On `done`: create new `narrative_draft_version` row with `version = current_version + 1`, `parent_version = current_version`, `generation_kind = 'section_regen'`. Replace the regenerated section's segments in `narrative_draft.segments`; leave other sections untouched.
6. Bump `narrative_draft.current_version`; emit `NARRATIVE_DRAFTED` event

The `narrative_draft.segments` jsonb stores all 4 sections together (Record-keyed by section_kind). Section regen replaces just one key. Version history preserves the full 4-section state at each version, enabling per-section rollback by diffing versions.

### UI rendering

`apps/web/src/lib/narrative/render.tsx`:

- `prose` → plain `<p>`
- `claim` → `<p>` followed by superscript footnote markers `[1] [2]` for each `citing_event`
- Footnotes → numbered list at section bottom with `<EventCitation eventId={id}/>` (card showing event kind + captured_at + first 100 chars of `payload.text`; clickable to navigate to event detail page)

Footnote numbering is per-section. PDF export (P7 territory): footnotes become endnotes per section; full event chain hash list on a separate "Audit appendix" page.

### Model + cost

**Sonnet 4.5.** ~6K output tokens per activity × $3/$15 per M → ~$0.10 per activity narrative. ~30 activities per claim → ~$3/claim. Override via `NARRATIVE_DRAFTER_MODEL`. Stub fallback emits deterministic placeholder text per section.

### Idempotency for streaming

`idempotency_key = sha256(activity_id + sorted(target_section_kinds) + prompt_version + model + client_request_id)` where `client_request_id` is a UUID the client generates per "Generate" button click.

If the client retries the same request (network hiccup, double-click, browser crash), server returns existing in-flight stream or completed draft — doesn't restart Anthropic.

For "I want a different draft" the consultant clicks "Regenerate" → new `client_request_id` → new idempotency key → new generation. Idempotency keys persist across retries; regeneration is a separate user intent.

### Failure modes

- **Stream cuts mid-generation:** server has been persisting segments to a builder. On stream error, write what we have to `narrative_draft` with `status='streaming'`, emit SSE `error`. Consultant sees "Resume" button; next request includes `segment_index_after_last_persisted` and the model continues.
- **Validation correction loop exhausted:** model can't get a segment right after 2 retries. Emit as `prose` (downgraded) with `validation_downgraded_count` flag. Consultant sees yellow flag.
- **Empty `clustered_events`:** activity has no events to cite. Refuse generation (400 with reason).
- **Anthropic outage / cancel mid-stream:** server cancels upstream cleanly; discards in-flight buffer if no segments completed; persists with `status='streaming'` if some did.
- **Section regen producing zero segments:** model returned but emitted nothing for the target section. Keep previous version's segments for that section; emit SSE `error` with `reason='no_segments_generated'`.
- **Client disconnects mid-stream:** server detects via SSE keepalive; cancels Anthropic; persists what we have. Next page-load shows partial draft with "Resume" affordance.

### Streaming infrastructure

New utility `apps/api/src/lib/sse.ts`:

```ts
export function startSSEStream(reply: FastifyReply): {
  send: (event: string, data: unknown) => void;
  close: () => void;
} {
  reply.raw.writeHead(200, {
    'Content-Type': 'text/event-stream',
    'Cache-Control': 'no-cache',
    'Connection': 'keep-alive',
    'X-Accel-Buffering': 'no'
  });
  return {
    send: (event, data) => reply.raw.write(`event: ${event}\ndata: ${JSON.stringify(data)}\n\n`),
    close: () => reply.raw.end()
  };
}
```

Used by Agent C's two endpoints. Server cancels Anthropic on client disconnect via `reply.raw.on('close', () => abortController.abort())`.

Client-side `apps/web/src/lib/sse-client.ts` wraps `EventSource` with auto-reconnect (5s backoff, 3 retries).

### Rate limiting

Anthropic enforces per-API-key rate limits. Per-(tenant, agent) token-bucket in `@cpa/agents/runtime/rate-limit.ts`:

```ts
async function rateLimitedAnthropicCall<T>(
  tenantId: string, agent: AgentName, fn: () => Promise<T>
): Promise<T> {
  const bucket = getBucket(tenantId, agent);  // e.g. 100 calls/min per (tenant, agent)
  await bucket.consume(1);
  return fn();
}
```

In-memory per-process. Per-tenant limits sized so a typical claim cycle fits in 5min, but a runaway loop tops out at ~$5/min across all agents.

## Section 7 — Testing strategy + risk register

### Testing layers per agent

| Layer | What it covers | Where it runs | Cost |
| --- | --- | --- | --- |
| Unit (node:test) | Pure-logic surface: prompt builders, validators, segment-correction loop, content-hash canonicalization, confidence-threshold mapping, SSE protocol formatting | CI on every PR, no DB, no Anthropic | Free |
| Integration (pg-boss + DB) | Job processors against real Postgres + stub classifier. Idempotency dedupe, event emissions, draft persistence, version history, RLS isolation | CI on every PR | Free |
| Stub end-to-end | Full pipeline (ingest → A → B → C) with `CLASSIFIER_IMPL=stub`. One Playwright spec exercises consultant happy path: synthesize register → accept → generate narrative → regen one section → save | CI on every PR | Free |
| Eval (golden datasets vs live model) | Quality regression detection: classifier F1, register cluster overlap, narrative structural compliance | CI on prompt-PRs only; nightly on `main` | ~$2/run |

### What stays out of automated testing

- Narrative prose quality (readability, tone) — pairwise human comparison during dogfood
- Activity naming creativity — pairwise human comparison
- Citation appropriateness — model can cite events that exist in the activity (validated automatically) but still wrong for the specific claim — human-judged

These manual reviews happen during staged rollout. Quality bars enforced via "kill the rollout" not "block the deploy".

### Open questions / TBDs for writing-plans

1. **Exact `section_kind` text** — `'experiments_and_results'` or split into `'experiments'` + `'results'`? Default: keep as one (matches AusIndustry treatment). Revisit only if consultant feedback says splitting helps.
2. **Confidence threshold calibration** — `0.85` and `0.70` are guesses. After dogfood week, examine override distribution and adjust. Action item, not architecture decision.
3. **`statutory_anchor` taxonomy** — currently `'s.355-25' | 's.355-30' | 'ineligible'`. AusIndustry has finer subdivisions (`s.355-25(2)(a)` etc). MVP: just core/supporting/ineligible. Finer in P7 if customers ask.
4. **Concurrent generation** — what happens when two consultants on the same firm hit "generate narrative" for the same activity at the same time? MVP: optimistic — both streams run, last-to-write wins on `narrative_draft.current_version`. Add advisory locking in P7 if it bites.
5. **Background batching for Agent A** — batch by "every 1min" or "every 25 events"? MVP: dual trigger — flush when 25 events OR 1min, whichever first. Tunable via env.
