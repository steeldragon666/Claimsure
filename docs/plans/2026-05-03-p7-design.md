## Section 2 — Theme A: Multi-cycle narrative + forensic metadata

### 2.1 — Goal

Carry an activity's narrative coherently across multiple FY claim cycles via the `proposed_id` chain established in P6, while satisfying the Body by Michael ART decision's evidentiary standard: each FY's hypothesis must be contemporaneously formed and dated; multi-cycle summaries cite prior segments, never paraphrase them.

### 2.2 — Schema additions (migration 0037)

```sql
-- Forensic provenance: every claim-bearing narrative segment carries an
-- immutable creation timestamp distinct from row's updated_at
ALTER TABLE narrative_segment
  ADD COLUMN first_recorded_at timestamptz NOT NULL DEFAULT now();

-- Activity-level forensic stamp: when was the hypothesis first formed?
-- Set on first save, immutable thereafter, drives Body by Michael defense.
ALTER TABLE activity
  ADD COLUMN hypothesis_formed_at timestamptz;

-- Backfill existing rows from earliest narrative_draft.created_at;
-- new rows enforce NOT NULL via trigger
UPDATE activity a
   SET hypothesis_formed_at = (
     SELECT MIN(nd.created_at) FROM narrative_draft nd WHERE nd.activity_id = a.id
   )
 WHERE a.hypothesis_formed_at IS NULL;

ALTER TABLE activity
  ALTER COLUMN hypothesis_formed_at SET NOT NULL;

-- Audit log entries on any attempted update of hypothesis_formed_at
CREATE OR REPLACE FUNCTION audit_hypothesis_formed_at_immutability()
RETURNS trigger AS $$
BEGIN
  IF OLD.hypothesis_formed_at IS DISTINCT FROM NEW.hypothesis_formed_at THEN
    INSERT INTO audit_log (
      tenant_id, kind, subject_kind, subject_id, payload, created_at
    ) VALUES (
      NEW.tenant_id,
      'HYPOTHESIS_FORMED_AT_IMMUTABILITY_VIOLATION',
      'activity', NEW.id,
      jsonb_build_object(
        'old_value', OLD.hypothesis_formed_at,
        'attempted_value', NEW.hypothesis_formed_at,
        'attempted_by', current_setting('app.current_user_id', true)
      ),
      now()
    );
    RAISE EXCEPTION 'hypothesis_formed_at is immutable; backdating attempts are audit-logged';
  END IF;
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER activity_hypothesis_formed_at_immutable
  BEFORE UPDATE ON activity
  FOR EACH ROW
  EXECUTE FUNCTION audit_hypothesis_formed_at_immutability();

-- Multi-cycle summary lives in narrative_draft.segments as a new
-- section_kind: 'multi_cycle_summary'. Segment payload references
-- prior NarrativeSegment rows by content_hash + activity_id + fy_label.
-- No new table needed for the summary itself — citation graph is data,
-- not structure.
```

### 2.3 — `proposed_id` continuity walk

When an activity is finalised in FY-N, `narrative_draft.activity_id` references the final activity row. The activity row carries `proposed_id` (from Agent B's draft register). To resolve "all prior FY narratives for this hypothesis":

```typescript
// packages/agents/src/multi-cycle.ts
async function walkProposedIdChain(
  rootProposedId: string,
  tenantId: string
): Promise<ActivityHistoryRow[]> {
  return await db.execute(sql<ActivityHistoryRow[]>`
    SELECT a.id AS activity_id,
           a.fy_label,
           a.hypothesis_formed_at,
           a.proposed_id,
           nd.id AS narrative_draft_id,
           nd.content_hash
      FROM activity a
      JOIN narrative_draft nd ON nd.activity_id = a.id
     WHERE a.tenant_id = ${tenantId}
       AND a.proposed_id = ${rootProposedId}
     ORDER BY a.fy_label ASC, a.hypothesis_formed_at ASC
  `);
}
```

The function returns all `(activity, narrative_draft)` pairs sharing a `proposed_id` across FYs, sorted chronologically. The narrative drafter consumes this list to build the prior-FY context block for the current-FY draft (Q5 default-on behaviour).

## Section 3 — Theme B: Prompt suggestion queue + GitHub App PR generation

### 3.1 — Goal

When the consultant identifies a problem with agent output (wrong inference, missed nuance, regulatory drift), the system curates the issue into actionable prompt-module changes and lands them as multi-file PRs the consultant reviews and merges. The platform owns the choreography; the consultant owns the merge decision.

### 3.2 — Schema additions (migration 0038)

```sql
CREATE TABLE prompt_suggestion (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  tenant_id uuid NOT NULL REFERENCES tenants(id),
  flagged_by_user_id uuid NOT NULL REFERENCES users(id),
  flagged_at timestamptz NOT NULL DEFAULT now(),
  source_kind text NOT NULL CHECK (source_kind IN (
    'consultant_flag',         -- consultant clicked "this output is wrong"
    'rif_event',               -- regulatory intelligence feed auto-flagged
    'contract_test_failure',   -- CI detected drift
    'reviewer_disposition'     -- multi-entity similarity reviewer flagged
  )),
  source_payload jsonb NOT NULL,                  -- context: which output, which row, which event
  affected_prompt_module text,                    -- e.g. 'draft-narrative@1.1.0'
  affected_section_kind text,
  issue_summary text NOT NULL,
  status text NOT NULL DEFAULT 'open' CHECK (status IN (
    'open','triaged','pr_drafted','pr_merged','dismissed'
  )),
  triage_classification text CHECK (triage_classification IN (
    'prompt_change','schema_change','code_change','no_action_needed'
  )),
  pr_id uuid REFERENCES prompt_suggestion_pr(id),
  resolved_at timestamptz,
  first_recorded_at timestamptz NOT NULL DEFAULT now()
);

CREATE TABLE prompt_suggestion_review (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  suggestion_id uuid NOT NULL REFERENCES prompt_suggestion(id),
  reviewer_user_id uuid NOT NULL REFERENCES users(id),
  reviewed_at timestamptz NOT NULL DEFAULT now(),
  disposition text NOT NULL CHECK (disposition IN (
    'approve_for_pr','request_more_info','dismiss','escalate_to_code_change'
  )),
  notes text
);

CREATE TABLE prompt_suggestion_pr (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  suggestion_id uuid NOT NULL REFERENCES prompt_suggestion(id),
  github_pr_number int NOT NULL,
  github_pr_url text NOT NULL,
  branch_name text NOT NULL,
  changed_files jsonb NOT NULL,                   -- array of {path, change_kind}
  created_at timestamptz NOT NULL DEFAULT now(),
  merged_at timestamptz,
  merge_commit_sha text
);
```

All tables RLS-protected; suggestions are visible across the tenant's review group, not just the flagger.

### 3.3 — GitHub App authentication

Theme B uses a dedicated GitHub App (not personal access token) for PR generation. Rationale: scoped permissions, revocable bot identity, audit trail tied to "cpa-platform-bot" rather than any human user.

**Required env vars:**
- `GITHUB_APP_ID` — numeric app ID
- `GITHUB_APP_PRIVATE_KEY` — PEM private key (loaded from secret manager, never committed)
- `GITHUB_APP_INSTALLATION_ID` — installation ID for the cpa-platform repo

**Permission scopes (least-privilege):**
- Contents: read + write (commit changes)
- Pull requests: read + write (create + comment)
- Metadata: read

The app cannot merge, cannot manage repo settings, cannot access other repos. Token rotation handled via JWT-based exchange (App generates installation tokens with 1-hour TTL).

### 3.4 — PR generation choreography

```typescript
// apps/api/src/routes/prompt-suggestions.ts
async function generatePRForSuggestion(
  suggestionId: string,
  reviewerUserId: string
): Promise<PromptSuggestionPr> {
  const suggestion = await loadSuggestion(suggestionId);
  if (suggestion.status !== 'triaged') {
    throw new Error('Suggestion must be triaged before PR generation');
  }

  // 1. Run prompt-suggestion-evaluate@1.0.0 to produce file change set
  const changeSet = await evaluateSuggestion(suggestion);

  // 2. Branch off main
  const branchName = `prompt-suggestion/${suggestion.id.slice(0,8)}`;
  await octokit.git.createRef({
    owner, repo, ref: `refs/heads/${branchName}`,
    sha: await getMainSha()
  });

  // 3. Multi-file commit (single commit, atomic)
  const tree = await octokit.git.createTree({
    owner, repo, base_tree: mainTreeSha,
    tree: changeSet.files.map(f => ({
      path: f.path,
      mode: '100644',
      type: 'blob',
      content: f.newContent
    }))
  });

  const commit = await octokit.git.createCommit({
    owner, repo, message: `prompt-suggestion: ${suggestion.issue_summary}`,
    tree: tree.data.sha,
    parents: [mainSha],
    author: { name: 'cpa-platform-bot', email: 'bot@cpa-platform.local' }
  });

  await octokit.git.updateRef({
    owner, repo, ref: `heads/${branchName}`, sha: commit.data.sha
  });

  // 4. Open PR with structured body (suggestion context + change set rationale)
  const pr = await octokit.pulls.create({
    owner, repo, head: branchName, base: 'main',
    title: `prompt-suggestion: ${suggestion.issue_summary.slice(0,72)}`,
    body: renderSuggestionPrBody(suggestion, changeSet, reviewerUserId),
    draft: false
  });

  // 5. Persist prompt_suggestion_pr row, update suggestion status
  return await persistPrAndUpdateSuggestion(suggestion.id, pr.data, branchName, changeSet);
}
```

The change set can span multiple files: prompt module text, Zod schema bumps, three-way-parity SQL CHECK updates, agent runtime version pins, regression test additions. The platform handles cross-file consistency (if a prompt module's output schema changes, the PR also bumps the consumer's version pin and adds a contract test).

## Section 4 — Theme C: Activity audit timeline + multi-entity comparison

### 4.1 — Goal

Render the full audit history of an activity in one place — every event, every narrative version, every reviewer disposition, every chain-anchor verification — so the consultant can answer "what changed and when" without grep'ing the database. Plus: side-by-side comparison of activities across the consultant's entity group, surfacing similarity scores from Theme D so DISR pattern-match exposure is visible before submission.

### 4.2 — No new tables

Theme C is read-side. It joins existing tables: `event`, `narrative_draft_version`, `audit_log`, `prompt_suggestion`, `multi_entity_similarity_score` (from Theme D). Migration 0039 is reserved but only used if benchmarks show timeline rendering needs additional indices.

### 4.3 — Timeline endpoint

```typescript
// apps/api/src/routes/audit-timeline.ts
GET /audit/activity/:activityId/timeline
Response: TimelineRow[]

interface TimelineRow {
  occurred_at: string;             // ISO timestamp
  kind: 'event' | 'narrative_version' | 'audit_log' | 'suggestion' | 'similarity_flag';
  subject_id: string;
  payload_summary: string;         // human-readable single-line summary

    first_recorded_at: string;

```

Each row carries a `chain_verified` boolean from running `verifyChain()` over the event's anchor segment. UI renders a green checkmark when verified, red X when broken (which would indicate database tampering — never expected, always actionable).

### 4.5.4 — Daily cron + classifier

```typescript
// tools/scripts/scrape-regulatory.ts
async function runDailyScrape(): Promise<void> {
  const sources = await db.execute(sql`
    SELECT * FROM regulatory_source WHERE enabled = true
       AND (last_polled_at IS NULL OR last_polled_at < now() - interval '1 hour' * fetch_interval_hours)
  `);

  for (const source of sources) {
    try {
      const newEvents = await fetchSource(source);
      for (const evt of newEvents) {
        // Insert raw event (idempotent on source_id + external_id)
        const eventRow = await insertOrSkipEvent(source.id, evt);
        if (!eventRow.classified_at) {
          // Classify via regulatory-classify@1.0.0
          const classification = await classifyEvent(eventRow);
          await persistClassification(eventRow.id, classification);
          // Dispatch webhooks if severity warrants
          if (classification.severity === 'high' || classification.severity === 'medium') {
            await dispatchToThemeB(eventRow, classification);  // suggestion queue
            await dispatchToThemeD(eventRow, classification);  // similarity corpus update / claim flag
          }
        }
      }
      await updateSourceStatus(source.id, 'success');
    } catch (err) {
      await updateSourceStatus(source.id, classifyError(err));
      // 7-day-stale data warning surfaces in /intelligence UI
    }
  }
}

// Cron schedule: daily at 03:00 AEST via existing pg-boss cron support
```
