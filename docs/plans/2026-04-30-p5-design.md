## Section 2 — Architectural components

### 2.1 `audit_log` table

New table, firm-scoped, no participation in the per-tenant hash chain.
Lives in its own migration (idx 22) and its own Drizzle schema file.

```sql
-- packages/db/migrations/0022_audit_log.sql
CREATE TABLE audit_log (
  id            uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  firm_id       uuid NOT NULL REFERENCES firm(id),
  kind          text NOT NULL,            -- AUDIT_KINDS member
  payload       jsonb NOT NULL,
  actor_user_id uuid REFERENCES "user"(id),
  created_at    timestamptz NOT NULL DEFAULT now()
);

CREATE INDEX audit_log_firm_id_created_at_idx
  ON audit_log (firm_id, created_at DESC);

ALTER TABLE audit_log ENABLE ROW LEVEL SECURITY;

CREATE POLICY audit_log_firm_isolation ON audit_log
  USING (firm_id = current_setting('app.current_firm_id')::uuid);
```

Rationale (from `0008`):

- **No hash chain in v1.** The chain invariant we market today is "no
  claimant evidence is silently mutated." Firm-scoped audit doesn't
  meet that bar. Adding a chain later is a column add + a backfill;
  it's reversible.
- **`firm_id` is GUC-keyed**, parallel to `app.current_tenant_id` for
  the existing `event` table. The middleware that sets the tenant GUC
  needs a parallel firm GUC (see pending decision below).
- **No `subject_tenant_id`.** Cross-tenant firm history queries don't
  need to join to claimants.

### 2.4 `apply-rules` endpoint

`POST /v1/expenditures/:id/apply-rules` is a separate endpoint, not a
flag on B10's preview. The brainstorm locked option (ii) of 5.4. The
shape:

```ts
// Request: empty body; rules come from the firm's mapping-rule set.
// Response:
{
  expenditureId: 'uuid',
  applied: [
    { ruleId: 'uuid', action: 'MAP_TO_ACTIVITY', targetId: 'uuid', eventId: 'uuid' },
    // EXPENDITURE_MAPPED event written for each `applied` entry
  ],
  skipped: [
    { ruleId: 'uuid', reason: 'no_match' },
    // Pending: rules with `flag_for_review` action — see "pending decisions" below
  ],
}
```

Semantics:

- **Idempotent on no-op**: replaying with no new matches returns
  `applied: []`. Already-mapped expenditures are not re-mapped.
- **Same-tx writes**: each match writes one `EXPENDITURE_MAPPED` event
  in the same transaction as the mapping row insert.
- **Race with preview**: B10's preview reads from `event` post-commit;
  no race because `apply-rules` doesn't touch the preview's read path.

### 2.5 Types relocation (`@cpa/types` → `@cpa/schemas`)

Per `0010` option C. `@cpa/schemas` already has zero workspace
dependencies and is imported by both `@cpa/db` and `@cpa/integrations`.
Adding type declarations there is a no-op in the dep graph.

Move list (verified after PR #4 lands; rebase may have shifted paths):

- `MappingRule` and `MappingRuleAction` from
  `packages/db/src/types/mapping-rule.ts` → `packages/schemas/src/mapping-rule.ts`.
- Inline duplicate in `packages/integrations/src/types.ts` removed;
  callsite swaps to `import type { MappingRule } from '@cpa/schemas'`.
- Where a moved type has a natural Zod form, define the Zod schema and
  export `type Foo = z.infer<typeof fooSchema>` to match the existing
  `@cpa/schemas` pattern.

A short note in `packages/db/README.md` documents the cycle constraint
(`@cpa/db` cannot import from `@cpa/integrations`) so future
contributors don't try to "fix" the duplication by closing the cycle.

### 2.6 Schema additions (Theme 1 columns)

Three columns, three migrations, all reversible at the schema level:

```sql
-- idx 18: claim.project_id (stays nullable forever — see locked Q4)
ALTER TABLE claim ADD COLUMN project_id uuid REFERENCES project(id);
CREATE INDEX claim_project_id_idx ON claim (project_id);
-- backfill from activity in same migration
UPDATE claim c SET project_id = a.project_id
FROM (
  SELECT DISTINCT ON (claim_id) claim_id, project_id
  FROM activity WHERE claim_id IS NOT NULL
  ORDER BY claim_id, project_id
) a WHERE c.id = a.claim_id;

-- idx 19: expenditure.claim_id (nullable; populated at map-time)
ALTER TABLE expenditure ADD COLUMN claim_id uuid REFERENCES claim(id);
CREATE INDEX expenditure_claim_id_idx ON expenditure (claim_id);

-- idx 20: expenditure_line.line_number (NOT NULL DEFAULT 1, unique within expenditure)
ALTER TABLE expenditure_line
  ADD COLUMN line_number integer NOT NULL DEFAULT 1;
CREATE UNIQUE INDEX expenditure_line_number_unique
  ON expenditure_line (expenditure_id, line_number);
```

Pre-flight backfill verification on idx 18 (must pass before the
migration lands in production):

```sql
-- expect 0 rows
SELECT claim_id, COUNT(DISTINCT project_id) AS projects
FROM activity
WHERE claim_id IS NOT NULL
GROUP BY claim_id
HAVING COUNT(DISTINCT project_id) > 1;
```

If any row returns, the data model has a deeper problem and the
migration halts — investigation precedes ship.

## Section 3 — Data flow, migrations, error handling, testing

### 3.1 Data flow

**`apply-rules` flow (#17, the keystone writer):**

```
client POST /v1/expenditures/:id/apply-rules
  └─> auth middleware sets app.current_tenant_id and app.current_firm_id
       └─> handler (apps/api/src/routes/expenditures.ts)
            ├─ load expenditure (RLS: subject_tenant_id)
            ├─ load firm's active mapping rules (RLS: firm_id)
            ├─ for each rule:
            │   ├─ evaluate match predicate
            │   ├─ if MAP_TO_ACTIVITY:
            │   │   ├─ INSERT mapping row
            │   │   └─ writeEvent(EXPENDITURE_MAPPED, subject_tenant=expenditure.tenant)
            │   └─ if FLAG_FOR_REVIEW: append to skipped (no event in v1)
            └─ commit tx; return { applied, skipped }
```

**`audit_log` write flow (#16, mapping rule lifecycle):**

```
B9's mapping-rule writer (POST/PATCH/DELETE on /v1/firms/:id/mapping-rules)
  └─ INSERT/UPDATE mapping_rule
       └─ writeAuditLog({ kind: MAPPING_RULE_*, firm_id, payload, actor_user_id })
            └─ INSERT audit_log (RLS: firm_id GUC must match)
```

**Theme 1 read-path simplification:**

```
Before:                          After (post #1):
GET /v1/claims?project_id=X      GET /v1/claims?project_id=X
  SELECT c.* FROM claim c          SELECT * FROM claim
  JOIN activity a ON ...           WHERE project_id = $1
  WHERE a.project_id = $1
  GROUP BY c.id
```

### 3.3 Error handling

- **Backfill ambiguity on #1.** Pre-flight SELECT is a hard gate.
  Failure → halt deployment, escalate. Documented as the single
  high-severity migration risk.
- **`audit_log` write without firm GUC set.** The RLS policy will
  reject the INSERT. The writer helper (`writeAuditLog()`) MUST throw a
  named error (`AuditFirmContextMissingError`) at the API layer, not
  let the database error bubble — the message must be debuggable.
- **`apply-rules` partial failure.** The endpoint is single-tx; any
  rule-evaluation failure rolls back all writes for that expenditure.
  Response 500 with structured error body listing the failed rule.
- **Hash chain regression on new kinds.** Adding `EXPENDITURE_MAPPED`
  and `EXPENDITURE_APPORTIONED` to `EVIDENCE_KINDS` requires the
  canonicaliser to handle them. A9's existing tests cover unknown
  kinds; new positive tests are required to lock the canonical shape.
- **GUC plumbing parity.** If `app.current_firm_id` is not set in a
  request that hits an `audit_log`-touching path, the request fails
  closed (RLS denies). No silent fallback to "any firm".

## Section 4 — Execution model and risk register

### 4.1 Execution model

Approach A — per-item dispatch, mirroring P4. Each item gets:

1. An **implementer** dispatch (writes the change, runs typecheck +
   tests locally, opens PR).
2. A **reviewer** dispatch (reads the diff against item's section in
   this design, applies fix-loop guidance to the implementer if
   anything's off).
3. **Fix loop** until reviewer says ship.

Worktrees are allocated by stream so contention on shared files is
minimal:

| Worktree | Stream | Items | Rationale |
|---|---|---|---|
| `p5a` | α denorm | #1, #2, #3, preview-rules cleanup | All three migrations + the two callsite updates land here. Single Drizzle schema author. |
| `p5b` | β audit_log + emissions | #4, #5, #6, #7, #14, #15, #16, #17 | Single-stream theme; one implementer keeps mental model. #13 also lands here. |
| `p5c` | γ-filters | #10, #11, #12 (after #1), #18, #19, #20 | Independent of α and β. |
| `p5d` | γ-architecture | #8, #9, #21, #22 | Types move + page-size const + CI trigger + retro doc. |

Branches use `p5-NNN-description` per `p5-roadmap.md`'s convention.
PR-per-item except where the design notes "ship with #X" (preview-rules
cleanup ships with #2 + #3; #5 ships with #4 since the AUDIT_KINDS
constant only makes sense alongside the new table).

### 4.2 Risk register

| # | Risk | Probability | Severity | Mitigation |
|---|---|---|---|---|
| 1 | Backfill ambiguity on #1 (claim has multi-project activities) | low | high | Pre-flight SELECT gate; halt + escalate if any rows return. Documented in idx 18 migration comment. |
| 2 | `audit_log` RLS regression — write path leaks across firms | low | high | Positive RLS test (#7) lands BEFORE writer #6 ships. Reviewer enforces. |
| 3 | `app.current_firm_id` GUC plumbing missed in some auth path | medium | medium | Audit middleware before #4 lands; add a parallel test asserting GUC is set in every authenticated request. |
| 4 | Hash chain regression from new event kinds | low | medium | Positive canonicaliser tests for each new kind (#14, #15) before writer #17 ships. |
| 5 | Test fixture drift on #3 (line-number swap) | medium | low | Half-day fixture cleanup baked into #3 estimate; seed produces an expenditure with two lines. |
| 6 | `apply-rules` races with B10's preview reads | low | medium | Same-tx semantics; preview reads `event` post-commit. Documented in #17's PR. |
| 7 | Type relocation drift caught only at boundaries | low | medium | `pnpm -r typecheck` in CI catches it; integration test exercises serialization. |
| 8 | Theme 2 cascade if keystone (#4) slips | medium | medium | Stream β is single-stream by design; reviewer flags any item that depends on #4 before #4 has landed. |
| 9 | PR #4 still open at P5 start | medium | high | P5 work cannot start until PR #4 lands on `main`. Hard gate; controller checks before spawning first dispatch. |

### 4.3 Success criteria

P5 is done when all of:

1. Every item in `0007`'s catalog is either landed or explicitly
   downgraded with a written reason (in the retrospective doc #22).
2. `audit_log` exists; `MAPPING_RULE_*` events emit on rule lifecycle;
   the positive RLS test passes.
3. `EXPENDITURE_MAPPED` and `EXPENDITURE_APPORTIONED` are first-class
   in canonicaliser + hash-chain tests; B10's preview is paired with a
   sibling writer at `POST /v1/expenditures/:id/apply-rules`.
4. `claim.project_id`, `expenditure.claim_id`, and
   `expenditure_line.line_number` are migrated; preview-rules.ts no
   longer uses the fiscal-year window or `ORDER BY id ASC`.
5. `MappingRule` lives in `@cpa/schemas`; no duplicate type
   declarations in `@cpa/db` or `@cpa/integrations`.
6. CI is green on `main`. No persistent skipped tests added during P5.
7. The P4 retrospective doc lists what was descoped vs delivered so P6
   inherits an accurate picture.

### 4.4 Rollback / abort criteria

P5 work is halted (and the offending item rolled back) if any of:

1. **Pre-flight SELECT on idx 18 returns rows.** Migration halts; the
   data anomaly is investigated before any further P5 work spawns. The
   broader stream α stalls until resolved.
2. **`audit_log` RLS test fails on a writer dispatch.** Writer #6
   (and any dependent: #16) cannot ship. Stream β halts; the policy is
   re-derived.
3. **Hash chain test regression on the new kinds.** #14 or #15 cannot
   ship without locking the canonical shape. Stream β chains stall on
   #17 until the canonicaliser is fixed.
4. **PR #4 surfaces a regression on `main` mid-P5.** All P5 dispatches
   pause; the controller re-runs the gate before resuming.

Rollback shape per stream:

- **Stream α**: each migration is reversible at the schema level
  (DROP COLUMN, DROP INDEX). Backfill rollback is "leave column null"
  — safe because the column stays nullable forever per locked Q4.
- **Stream β**: `audit_log` table can be dropped; the writer helper
  becomes a no-op. Reserved kinds revert to their B9 state (dead code
  in `EVIDENCE_KINDS`) — exactly the pre-P5 condition.
- **Stream γ**: filters are pure handler additions; revert is `git
  revert`. Type relocation is reversible by re-inlining the
  declarations.
